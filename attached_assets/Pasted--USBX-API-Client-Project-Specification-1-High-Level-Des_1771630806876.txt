# USBX API Client - Project Specification

## 1. High-Level Description

### What It Does
USBX is a full-featured API testing client, similar to Postman or Hoppscotch. It allows developers to:
- Create, organize, and execute HTTP/REST API requests
- Test WebSocket connections
- Manage collections of requests in folders
- Use environment variables for different deployment contexts
- Run test assertions against API responses
- Import/export collections (Postman format)
- Generate and view API documentation

### Target Users
- Backend and frontend developers testing APIs
- QA engineers validating API contracts
- DevOps teams debugging integrations
- Technical teams sharing API collections

### What Makes It "Postman-like"
- **Collections**: Hierarchical organization of requests into named collections and folders
- **Environments**: Named sets of variables (dev, staging, prod) with `{{variable}}` interpolation
- **Request Builder**: Full HTTP method support with headers, query params, body editors (JSON, form-data, raw, GraphQL)
- **Authentication**: Multiple auth types (Basic, Bearer, OAuth 2.0, AWS Signature, API Key, Digest, NTLM, Hawk)
- **Response Viewer**: Syntax-highlighted response body, headers, timing, size metrics
- **History**: Automatic logging of executed requests
- **Testing**: Assertions and scripts (pre-request, post-response) with Postman-compatible test syntax
- **Import/Export**: Postman Collection v2.1 import support

## 2. Tech Stack

### Frontend Framework
- **React 19** with TypeScript
- **TailwindCSS 4** for styling (utility-first, no custom CSS files)
- **Radix UI** primitives (dialogs, dropdowns, tabs, tooltips, etc.)
- **shadcn/ui** component patterns

### State Management
- **Zustand** with persistence middleware
- Async store operations with loading states
- UI state (theme, active IDs) persisted to localStorage

### Networking Layer
- **Browser mode**: Native `fetch()` API (subject to CORS)
- **Electron mode**: Axios via IPC to main process (bypasses CORS)
- Custom `HttpClient` class with environment detection

### Persistence
- **Web**: Browser `localStorage` via `LocalDataService`
- **Electron**: Same localStorage (could extend to file system)
- **Data Service abstraction**: Interface allows swapping to remote API backend

### Code Editor / Syntax Highlighting
- **CodeMirror 6** with language modes: JSON, JavaScript, XML, HTML, CSS, Markdown

### Build Tools
- **Vite 6** for web development and production builds
- **electron-vite** for Electron builds
- **Vitest** for unit testing
- **Playwright** for E2E testing
- **ESLint** for linting

### Package Manager
- **npm** (uses `package-lock.json`)
- Single package (not a monorepo)

### Electron Stack
- **Electron 37** for desktop app
- **Main process**: HTTP client with proxy/SSL support, WebSocket handling, OAuth 2.0 flows, secure token storage
- **Preload script**: IPC bridge exposing `window.electronAPI`
- **Renderer**: Same React app as web, with feature detection for Electron APIs

## 3. Project Structure (Conceptual)

### Web App (`src/renderer/`)
The primary React application:
- **Entry point**: `index.html` → `main.tsx` → `App.tsx`
- **Main view**: `APIClient.tsx` (tabbed interface with request builder, sidebar, response panel)
- **Components**: UI components, feature components (auth, collections, environments, tests)
- **Services**: Data abstraction layer (`DataService`, `LocalDataService`, `RemoteDataService`)
- **Store**: Zustand store with collections, requests, environments, history, settings
- **Utils**: HTTP client, test engine, bulk runner, mock server engine
- **Types**: TypeScript interfaces for all domain models

### Electron App (`src/main/`, `src/preload/`)
Desktop-specific code:
- **Main process** (`src/main/`): Native HTTP client (Axios), proxy/SSL management, cookie handling, WebSocket client, OAuth 2.0 handler, secure storage
- **Preload** (`src/preload/`): IPC bridge exposing safe APIs to renderer via `contextBridge`
- **Shared types** (`src/types/`): IPC message types, HTTP request/response types

### Shared Logic
- The renderer code (`src/renderer/`) is 100% shared between web and Electron
- Feature detection (`window.electronAPI`) enables/disables Electron-specific features
- Same React components render in both environments

## 4. Web App Details

### Development Command
```bash
npm run dev
```
Uses Vite with config file `vite.web.config.ts`.

### Dev Server Port
**Port 3000** (configurable via Vite config)

### Backend Requirements
**None** — The web app is fully client-side:
- Data stored in browser `localStorage`
- HTTP requests made directly from browser (subject to CORS)
- No server-side API required

### Environment Variables
**None required** for basic operation. The app is self-contained.

Optional future variables (for remote data service):
- `VITE_API_BASE_URL` — Backend API endpoint
- `VITE_AUTH_TOKEN` — JWT for backend auth

### Routing Approach
**No router** — Single-page application with tab-based navigation:
- Tabs for open requests and collections
- Sidebar navigation between Collections, Environments, History, Mock Servers
- Modal dialogs for settings, import, save operations

## 5. Electron App Details

### What Electron Adds
- **CORS bypass**: HTTP requests via main process (Axios) bypass browser CORS restrictions
- **Full response headers**: Access to all headers including those browsers hide
- **Proxy support**: HTTP/HTTPS/SOCKS4/SOCKS5 with authentication
- **SSL/TLS**: Custom certificates, self-signed cert handling, client certificates
- **Cookie management**: Proper cookie jar with domain/path handling
- **WebSocket connections**: Native WebSocket client without browser limitations
- **OAuth 2.0**: System browser popup for authorization code flow
- **Native menus**: Application menu with keyboard shortcuts
- **File system**: Native save/open dialogs for import/export
- **Secure storage**: Encrypted token storage (via keytar or similar)
- **NTLM authentication**: Windows integrated auth (requires native modules)

### Storage Approach
Same `localStorage` used in web mode. The data service abstraction allows future migration to file-based or SQLite storage.

### UI Approach
The Electron app embeds the **same React UI** as the web version. No Electron-only UI components — just enhanced capabilities via IPC.

### Feature Detection
```typescript
const isElectron = typeof window.electronAPI?.httpRequest === 'function';
```
Components check this flag to show/hide Electron-only features (e.g., NTLM auth shows "Electron only" badge in web mode).

## 7. Non-Goals / Simplifications

For a first recreation, these can be omitted or simplified:

### Skip Entirely (Electron-Only)
- NTLM authentication
- System proxy detection
- Custom SSL certificates / client certificates
- Secure encrypted token storage
- Native file dialogs (use browser file picker instead)
- Native application menus

### Simplify
- **Mock Server**: Currently simulated in-browser. Can be a simple UI without actual server.
- **WebSocket**: Can start with Electron-only or skip entirely for web-first.
- **OAuth 2.0**: Can start with token input only, skip authorization code flow.
- **Bulk Runner**: Can start with sequential-only, skip parallel execution.
- **Test Scripts**: Can start with assertions-only, skip JavaScript scripting.
- **Import/Export**: Start with Postman import only, skip other formats.
- **Request Templates**: Can be a later addition.
- **Documentation Editor**: Can be a later addition.
- **Network Debug Panel**: Development tool, not essential.

### Data Layer
- Start with `localStorage` only
- The `RemoteDataService` interface exists but doesn't need implementation initially

### Testing
- Unit tests (Vitest) and E2E tests (Playwright) exist but aren't required to recreate the app

**Prompt:**

Create a Postman-like API testing client called "USBX" using React, TypeScript, TailwindCSS, and Vite.

**Start Simple:**
1. First: Request builder + Send button + Response viewer
2. Then: Collections sidebar
3. Then: Environments with variable interpolation
4. Then: History
